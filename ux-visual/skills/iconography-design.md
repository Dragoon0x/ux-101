# Iconography Design
**Purpose:** Design icons that communicate instantly and consistently — the visual shorthand that makes interfaces scannable and learnable.

Icons serve two purposes in interfaces: they communicate meaning (the trash can means delete) and they save space (a symbol fits where a word doesn't). The best icons do both. The worst icons do neither — decorative elements that add visual weight without adding comprehension.

## The Recognition Test
An icon that needs a label to be understood is decoration, not communication. Test icon comprehension by showing users the icon without its label and asking what it means. If fewer than 80% of users correctly identify the meaning, the icon needs a label. And that's fine — many actions are better communicated by icon-plus-label than by icon alone.

## The Consistency System
Icons in a product should look like they were designed by the same person in the same sitting. This means consistent: stroke weight, corner radius, grid alignment, visual weight, level of detail, and metaphor style. Mixing outlined and filled icons, or detailed and minimal icons, creates visual discord.

## The Sizing Architecture
Icons should be designed on a consistent grid (typically 16px, 20px, 24px, 32px) and exported at specific sizes with appropriate optical adjustments. An icon designed at 24px and scaled to 16px will have blurry strokes. Icons need to be optimized for each target size.

## When to use
When designing an icon system for a product. When existing icons are inconsistent. When adding new icons to an existing set. When icons are causing user confusion (misinterpreted actions).
