# Interaction Design Patterns
**Purpose:** Apply proven interaction patterns to recurring design problems — not blindly, but with understanding of why each pattern works and when it doesn't.

Interaction patterns are solutions that have been tested by millions of users across thousands of products. Using them is not lazy — it's respectful of user expectations. Users bring learned behaviors from every product they've ever used. When your product uses familiar patterns, users can focus on their goals instead of learning your interface.

## The Pattern Selection
For any interaction problem, there are typically 3-5 established patterns that could work. The selection depends on context: how much data is involved, how frequently the action occurs, how error-prone the action is, and what device the user is on. A date picker works for booking a hotel room. A text input works for entering a birthdate. Same data type, different contexts, different patterns.

## The Deviation Decision
Sometimes no existing pattern fits. Before inventing a new one, verify that the problem is genuinely novel — most "new" interaction problems have been solved before in a different domain. If the problem truly requires a new pattern, invest in testing it. Novel patterns carry a learning cost that established patterns don't.

## The Consistency Principle
Within a product, similar actions should use similar patterns. If one form uses inline validation and another uses submit-then-validate, users can't predict the behavior. Internal consistency reduces the cognitive load of using the product.

## When to use
Every interaction design decision. When choosing between alternative patterns for the same problem. When auditing an existing product for interaction consistency. When designing for a new platform or device.
