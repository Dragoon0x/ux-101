# User Flow Design
**Purpose:** Design the paths users take through a product — mapping, optimizing, and testing the sequences of steps that lead from intent to outcome.

A user flow is a hypothesis about how a user will accomplish a goal. The design question is not "what screens do we need?" — it's "what's the minimum number of decisions the user needs to make to get from wanting to doing?" Every additional step, every additional decision, every additional piece of information requested is friction that reduces completion.

## The Happy Path
Design the happy path first — the most common scenario where everything goes right. This path should be as short and frictionless as possible. It's the experience 80% of users will have, and it should be optimized for them.

## The Edge Cases
Then design the edge cases — what happens when data is missing, when the user changes their mind, when an error occurs, when the user is returning to complete something they started earlier. Edge cases are where most usability problems live because they're often designed as afterthoughts.

## The Return Path
Users don't always complete flows in one sitting. The return path — how a user picks up where they left off — is often overlooked. Save state. Show progress. Make it easy to resume. The return path is especially critical for mobile users who frequently interrupt and resume.

## When to use
When designing any multi-step process. When optimizing conversion funnels. When users report that a process is too long or confusing. When simplifying an existing flow.
