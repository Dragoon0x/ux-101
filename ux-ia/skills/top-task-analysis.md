# Top Task Analysis
**Purpose:** Identify the tasks that matter most to users — because information architecture should be optimized for the top 5-10 tasks, not for organizational completeness.

Gerry McGovern's Top Tasks methodology reveals that in any product, a small number of tasks account for the vast majority of user intent. Typically, the top 5 tasks represent 25-30% of all user activity. The top 20 represent 60-70%. The remaining tasks are 'tiny tasks' that collectively matter but shouldn't drive architectural decisions.

## The Method
List every task users might attempt (aim for 60-100 tasks). Survey users: ask them to vote for their top 5 from the full list. Analyze: tasks will follow a long-tail distribution — a few tasks get massive vote share, most get very few. The top tasks are your architectural priorities. Design navigation, labeling, and page hierarchy to optimize for them.

## The Organizational Trap
Organizations naturally want to give equal visibility to everything they offer. Top task analysis gives you the evidence to push back: "70% of users come here for these 5 things. Our architecture should make those 5 things effortless, even if it means other things require an extra click."

## When to use
When redesigning navigation. When prioritizing which flows to optimize. When the product has grown organically and the architecture no longer reflects user priorities.
