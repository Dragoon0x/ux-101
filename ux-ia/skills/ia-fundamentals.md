# IA Fundamentals
**Purpose:** Organize information so that humans can find, understand, and use it — the discipline that determines whether a product feels navigable or overwhelming.

Information architecture is invisible when it works and catastrophic when it doesn't. A well-organized product feels simple even when it's complex. A poorly organized product feels confusing even when it has few features. The architecture is the difference — the underlying structure that determines how content is grouped, labeled, and connected.

## The Organization Schemes
There are five ways to organize anything: alphabetically (good for known items), chronologically (good for time-based content), geographically (good for location-based content), by topic (good for exploration), and by task (good for goal-oriented users). Most products use a combination. The skill is knowing which scheme serves which user need and blending them coherently.

## The Labeling Problem
Labels are the interface to the architecture. A perfectly organized system with confusing labels is an unusable system. Labels should use the user's language, not the organization's internal terminology. "Account Settings" vs "Preferences" vs "My Profile" — the right label depends on what users call it, which you can only learn through research.

## The Depth vs Breadth Tradeoff
A navigation with 20 top-level items is overwhelming. A navigation with 3 top-level items but 5 levels of depth is disorienting. The research is clear: broader, shallower hierarchies outperform narrow, deep ones for most use cases. Users can scan 7-9 items faster than they can navigate 4 levels of nesting.

## When to use
When designing a new product. When existing navigation isn't working (high bounce rates, support tickets about finding things). When the product's content has outgrown its original organization. When user research reveals navigation confusion.
